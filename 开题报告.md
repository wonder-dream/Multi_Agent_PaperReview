## 三、相关研究工作

为明确本项目的切入点和创新性，简要梳理以下三个方面的已有工作：

1. **自动审稿辅助与论文完整性分析**  
   PeerRead 数据集 [7] 开启了利用 NLP 分析审稿文本的研究，但多数工作集中在直接预测”接受/拒绝”分数，缺乏对论文内容结构化分析和细粒度检查的能力。基于模板的简单分类系统虽能生成粗略评语，却难以逐一检查数据集多样性、Baseline 对比、消融实验等具体维度。

2. **科学信息抽取与知识图谱构建**  
   SciBERT [1] 和 SciERC [2] 等工作的出现显著提升了领域内 NER 与关系抽取的性能，可提取模型、数据集、指标等实体及彼此间的语义关系。然而这些抽取结果多停留在构建静态知识图谱，尚未被系统性地用于生成有据可依的完整性检查清单。

3. **多 Agent 与工具增强的大语言模型应用**  
   ReAct [5] 等推理框架展示了让 Agent 在”思考-行动-观察”循环中使用外部工具解决问题的高效性。近期系统如 Chameleon [6] 进一步通过模块化组合增强了LLM的推理与工具调用能力。但将这种 Agent 范式与自研精细 NLP 模块结合，针对科研文献完整性分析这一垂直场景进行系统设计与验证的实践仍属少见。

综上，本项目在”深度科学语义抽取”与”Agent 驱动的工作流编排”交叉地带进行探索，既依托自研 NLP 模型保障语义理解的可靠性，又借助 Agent 框架实现灵活、可反思的完整性分析流程。

## 四、选题背景与问题定义

学术论文审稿是科研质量控制的核心环节，当前面临审稿周期长、审稿人认知负荷高、实验复现性检查困难等挑战。多数现有辅助工具仅停留在关键词匹配或单一 NLP 分类任务，无法提供基于全文学术要素的结构化检查。

本项目定义的新应用场景为：**以自研 NLP 技术为语义理解核心、以 Agent 编排为任务调度框架的审稿人辅助系统**——面向会议/期刊审稿人，在正式审稿前自动生成论文技术画像与结构化完整性检查清单，帮助审稿人快速把握论文的技术要素和潜在缺口，降低认知负荷。重点解决四类问题：

1. 论文领域/方法类型的自动分类；
2. 科研实体（模型、数据集、指标等）与关系的结构化抽取；
3. 基于抽取事实的论文画像与完整性检查清单生成（数据集多样性、Baseline对比、消融实验、代码开源等维度）；
4. 基于语义检索的相关工作推荐与重复性检测；
5. 将上述 NLP 能力通过 Agent 编排为可反思、可纠错的完整性分析流程。

## 五、研究/构建内容与目标

### 总体目标

实现一个以 **4 个自研 NLP 模块** 为语义基础、以 **Agent 编排框架** 为调度中枢的科研文献智能分析原型系统，面向审稿人输出论文技术画像与完整性检查清单、科研知识图谱和相似论文推荐。

### 具体内容（应用型，组合 4 种 NLP 技术）

1. **NLP 模块一：科研文本分类器**  
   基于 SciBERT 微调，实现领域（多标签）、方法类型（单标签）二维分类，为 Agent 后续分析提供领域上下文。

2. **NLP 模块二：科研信息抽取引擎**  
   采用 SciBERT + BiLSTM + CRF 抽取 MODEL、DATASET、METRIC 等实体，并基于 Span-based 联合抽取获取 EVALUATED_ON、ACHIEVES 等关系三元组，为完整性检查提供事实基础。关系抽取目标 F1 > 50%（对齐 SciERC 数据集当前 SOTA ~51%）。针对 512-token 长文档限制，采用 LLM 全文预处理 + SciBERT 精确验证的混合流水线（参考 RATE 2025 架构，F1 提升可达 37 个百分点）。

3. **NLP 模块三：科研文本摘要与完整性检查器**  
   融合抽取式（TextRank + 句子向量）与生成式（BART-large / Flan-T5）方法，生成结构化摘要。基于抽取事实生成**论文画像**（使用了什么数据集/模型/指标/方法）和**完整性检查清单**（数据集多样性、Baseline 对比、消融实验、代码开源等维度），不做 Accept/Reject 判断。

4. **NLP 模块四：语义检索与匹配引擎**  
   以 SPECTER 或 SciBERT-NLI 编码论文向量，基于 FAISS 实现相似论文检索、重复性检测、引用推荐。

5. **Agent 智能编排层**  
   设计 Coordinator、Classifier、Extractor、Analyst、Retriever、Reviewer 六个角色 Agent，基于 ReAct 循环按依赖关系调度上述 NLP 模块和外部工具（论文库 API）。各 Agent 通过共享工作记忆同步中间结果，Coordinator 负责任务分解与并行调度，Reviewer 负责汇总报告并执行反思自检（逐维度确认检查清单是否遗漏关键项）。Agent 层不介入语义计算——所有语义理解由 NLP 模块完成，Agent 仅负责任务编排与结果校验。

6. **系统集成与可视化**  
   打通 PDF 解析（GROBID）、图表提取、FastAPI 后端与 React+D3.js 前端，实现端到端可交互原型。

> 说明：NLP 模块作为松耦合服务，以标准函数接口形式暴露给 Agent。Agent 层仅负责任务分解、依赖调度（串行/并行）与结果校验，所有语义理解由 NLP 模块独立完成。该设计的代价是端到端优化较困难（错误可能来自任一层的级联传播），优势是各层职责清晰、模块可独立替换。

## 六、技术路线与实施方案

### 6.1 技术框架

采用“数据与工具层 → NLP 核心能力层 → Agent 智能编排层 → 应用表现层”四层架构，各层通过轻量 API 通信。

### 6.2 关键技术与工具

- **预训练基座**：SciBERT [1]、SPECTER2 [3]、BART-large / LED、Flan-T5
- **深度学习环境**：PyTorch + Hugging Face Transformers
- **抽取算法**：BiLSTM-CRF、Span-based 联合实体关系抽取 [2]
- **检索**：Sentence-Transformers + FAISS
- **Agent 实现**：自研轻量 ReAct 循环 [5]，兼容 OpenAI 风格的 Function Calling
- **记忆存储**：短期记忆（结构化状态 JSON + 上下文窗口），长期记忆（Neo4j 图库 + FAISS 向量库）
- **系统构建**：FastAPI、React、D3.js
- **PDF 处理**：GROBID（参考文献）、MinerU / Marker（正文/表格/公式）
- **LLM API**：Gemini 3.1 Pro（长文档阅读，$2/1M token）、Claude Opus 4.7（事实校验，幻觉率 36%）

### 6.3 数据计划

- 分类数据：arXiv 元数据（20K+）、PeerRead [7]（3K+）
- 信息抽取数据：SciERC + Re² (Zhang et al. 2025, 19.9K 篇保证论文-审稿版本一致性)
- 端到端评估：PeerRead 论文数据 + Re²
- 摘要数据：SciTLDR [4]、PubMed 摘要（10K+ 对）

所有数据均可通过学术途径合法获取，预处理包括文本清洗、分句分块、图表 caption 提取、科学符号归一化。

### 6.4 实验环境

- GPU：至少 1 块 NVIDIA A100 或同等算力（学校集群/云平台）
- OS：Linux；开发语言：Python + JavaScript

## 七、评测方法与预期成果

### 7.1 评测指标

- **NLP 模块级**  
  - 分类：Macro-F1、Accuracy；NER：Span-F1（目标 > 78%）；关系抽取：Micro-F1（目标 > 50%，对齐 SciERC SOTA ~51%）  
  - 摘要：ROUGE-1/2/L、BERTScore、FactCC（事实一致性检查）；检索：Recall@K、MRR、NDCG@10
  - 辅助评估：LLM-as-Judge（补充自动指标与人类判断相关性低的问题）、消融实验（量化 LLM 预处理对下游精度的贡献）
- **Agent 系统级**  
  - 任务完成率、正确工具选择率、引入反思后检查清单覆盖维度提升幅度（通过人工评分衡量）
- **人工评估**  
  邀请 3-5 名研究生对生成的完整性检查清单按五维 Likert 量表评分：  
  **覆盖完整性**（是否覆盖数据集、Baseline、消融、代码等关键维度）、**项准确性**（每项判断是否符合论文实际）、**建议实用性**（建议是否具体可操作）、**领域适配度**（检查标准是否适配该子领域）、**可读性**（语言是否清晰规范）。

### 7.2 预期成果

1. 可正常运行、交互的原型系统；
2. 四个独立评测达到预设指标的 NLP 模块；
3. 一套面向科研文献完整性分析的 Agent 编排工作流；
4. 输出示例：论文画像与完整性检查清单、知识图谱、相似论文推荐；
5. 完整的技术文档与答辩演示材料。

## 八、创新点与可行性分析

### 创新点

1. **深度语义与编排调度的交叉**：将自研科学文本抽取、生成、检索等 NLP 模块集成于 ReAct 驱动的 Agent 编排框架中，通过共享记忆与反思校验形成完整性分析闭环。与纯 LLM-based 审稿系统（如 PaperReview.ai）不同，本系统的语义理解由可独立评测的专用 NLP 模块完成，而非黑箱 LLM 端到端生成。
2. **基于事实的检查清单生成**：所有检查项以抽取出的实体与关系为锚点（如”在2个数据集上验证，该领域标准≥3个”），不做主观 Accept/Reject 判断。与生成式审稿系统（输出”这篇论文实验不够充分”的叙述文本）不同，本系统输出的是逐项可验证的结构化检查条目（”识别到2个数据集，未发现消融实验段落”），审稿人可快速逐条确认或驳回。
3. **反思记忆设计**：通过维度覆盖自检（是否遗漏数据集、Baseline、消融、代码开源、伦理声明等关键维度），使系统在每次分析结束后强制执行一轮完整性校验。该机制类似飞行检查清单——目标是确保覆盖完整性而非”学会判断论文好坏”，不依赖外部反馈信号即可产生价值。

### 可行性

- **团队能力**：成员已完成 NLP 课程学习，熟悉 transformers 微调和基础 Agent 开发。
- **资源基础**：所需预训练模型与数据集均为开源，GPU 可向学校或云平台申请。
- **时间风险**：四个 NLP 模块可并行推进，Agent 开发与系统集成时间窗口明确，可控性强。

### 风险预案

| 风险 | 应对 |
|------|------|
| 抽取标注数据量小 | 弱监督（远程监督 + 规则标注）为主，少量人工精标；探索 GPT-4 辅助标注 |
| 长论文超 token 限制 | (1) LLM 全文预处理生成语义分块（推荐）；(2) 滑动窗口分块 + 段落边界感知切割 + 128 token 重叠（降级） |
| Agent 调用链延迟高 | 模块结果缓存 + 独立 Agent 并行调度；最大推理步数限制（5-7 步） |
| 生成内容偏离原文事实 | 生成前注入抽取事实约束，禁止自由编造；摘要输出过 FactCC 事实一致性检查 |
| 多模块级联误差传播 | 模块间置信度阈值过滤；Inspector Agent 校验关键输出；各模块独立降级路径 |
| Agent 工具调用幻觉 | 确定性工具路由（代码映射，LLM 不输出工具名）；每工具独立 circuit breaker |
| LLM API 不可用 | LLM 预处理为可选增强路径，核心功能不依赖 API，均有传统方案降级 |
