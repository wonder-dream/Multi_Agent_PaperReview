## 三、相关研究工作

为明确本项目的切入点和创新性，简要梳理以下三个方面的已有工作：

1. **自动审稿与论文元分析**  
   PeerRead 数据集 [7] 开启了利用 NLP 分析审稿文本的研究，但多数工作集中在直接预测“接受/拒绝”分数，缺乏对论文内容深度推理和结构化意见生成的能力。基于模板或简单分类的组合系统虽能生成粗略评语，却难以捕捉细粒度科研实体和实验逻辑。

2. **科学信息抽取与知识图谱构建**  
   SciBERT [1] 和 SciERC [2] 等工作的出现显著提升了领域内 NER 与关系抽取的性能，可提取模型、数据集、指标等实体及彼此间的语义关系。然而这些抽取结果多停留在构建静态知识图谱，尚未被系统性地用于辅助审稿决策。

3. **多 Agent 与工具增强的大语言模型应用**  
   ReAct [5] 等推理框架展示了让 Agent 在“思考-行动-观察”循环中使用外部工具解决问题的高效性。近期系统如 Chameleon [6] 进一步通过模块化组合增强了LLM的推理与工具调用能力。但将这种 Agent 范式与自研精细 NLP 模块结合，针对科研文献审稿这一垂直场景进行系统设计与验证的实践仍属少见。

综上，本项目在“深度科学语义抽取”与“Agent 驱动的工作流编排”交叉地带进行探索，既依托自研 NLP 模型保障语义理解的可靠性，又借助 Agent 框架实现灵活、可反思的审稿推理流程。

## 四、选题背景与问题定义

学术论文审稿是科研质量控制的核心环节，当前面临审稿周期长、审稿人认知负荷高、实验复现性检查困难等挑战。多数现有辅助工具仅停留在关键词匹配或单一 NLP 分类任务，无法为审稿人提供基于全文深度理解的决策建议。

本项目定义的新应用场景为：**以自研 NLP 技术为语义理解核心、以多 Agent 协作为任务调度框架的科研论文智能审稿辅助**。重点解决五类问题：

1. 论文领域/方法/质量的自动多维分类；
2. 科研实体（模型、数据集、指标等）与关系的结构化抽取；
3. 有据可依的结构化摘要与审稿意见初稿生成；
4. 基于语义检索的相关工作推荐与重复性检测；
5. 将上述 NLP 能力通过 Agent 编排为可反思、可纠错的完整审稿流程。

## 五、研究/构建内容与目标

### 总体目标

实现一个以 **4 个自研 NLP 模块** 为语义基础、以 **多 Agent 协作框架** 为调度中枢的科研文献智能分析原型系统，输出结构化审稿报告、科研知识图谱和相似论文推荐。

### 具体内容（应用型，组合 4 种 NLP 技术）

1. **NLP 模块一：科研文本分类器**  
   基于 SciBERT 微调，实现领域（多标签）、方法类型（单标签）、质量初筛（二/三分类）三维分类，为 Agent 任务分配提供依据。

2. **NLP 模块二：科研信息抽取引擎**  
   采用 SciBERT + BiLSTM + CRF 抽取 MODEL、DATASET、METRIC 等实体，并基于 Span-based 联合抽取获取 EVALUATED_ON、ACHIEVES 等关系三元组，构建论文知识图谱基础。

3. **NLP 模块三：科研文本摘要与生成器**  
   融合抽取式（TextRank + 句子向量）与生成式（BART-large / Flan-T5）方法，生成结构化摘要（背景、贡献、方法、实验、结论、局限），并结合抽取事实生成审稿意见草稿。

4. **NLP 模块四：语义检索与匹配引擎**  
   以 SPECTER 或 SciBERT-NLI 编码论文向量，基于 FAISS 实现相似论文检索、重复性检测、引用推荐。

5. **多 Agent 智能编排层**  
   设计 Coordinator、Classifier、Extractor、Analyst、Retriever、Reviewer 六个 Agent，基于 ReAct 循环调用上述 NLP 模块和外部工具（论文库 API、代码沙箱），通过短期/长期记忆与反思机制完成报告生成与自纠。

6. **系统集成与可视化**  
   打通 PDF 解析（GROBID）、图表提取、FastAPI 后端与 React+D3.js 前端，实现端到端可交互原型。

> 说明：NLP 模块作为松耦合服务，以标准函数接口形式暴露给 Agent，Agent 仅负责任务分解、调度与结果校验，不介入语义计算过程，保障 NLP 能力在系统中的核心地位。

## 六、技术路线与实施方案

### 6.1 技术框架

采用“数据与工具层 → NLP 核心能力层 → Agent 智能编排层 → 应用表现层”四层架构，各层通过轻量 API 通信。

### 6.2 关键技术与工具

- **预训练基座**：SciBERT [1]、SPECTER [3]、BART-large、Flan-T5
- **深度学习环境**：PyTorch + Hugging Face Transformers
- **抽取算法**：BiLSTM-CRF、Span-based 联合实体关系抽取 [2]
- **检索**：Sentence-Transformers + FAISS
- **Agent 实现**：自研轻量 ReAct 循环 [5]，兼容 OpenAI 风格的 Function Calling
- **记忆存储**：短期记忆（结构化状态 JSON + 上下文窗口），长期记忆（Neo4j 图库 + FAISS 向量库）
- **系统构建**：FastAPI、React、D3.js
- **PDF 处理**：GROBID、PyMuPDF

### 6.3 数据计划

- 分类数据：arXiv 元数据（20K+）、PeerRead [7]（3K+）
- 信息抽取数据：SciERC
- 摘要数据：SciTLDR [4]、PubMed 摘要（10K+ 对）
- 端到端评估：PeerRead 审稿报告（3K+）

所有数据均可通过学术途径合法获取，预处理包括文本清洗、分句分块、图表 caption 提取、科学符号归一化。

### 6.4 实验环境

- GPU：至少 1 块 NVIDIA A100 或同等算力（学校集群/云平台）
- OS：Linux；开发语言：Python + JavaScript

### 6.5 分阶段实施步骤

|------|----------|------------|

| **阶段2（第3-5周）** | 4 个 NLP 模块独立训练与评测达标；Agent 框架完成 ReAct 循环与工具调用对接；打通一条“全文→审稿草稿”流程 | 各模块指标达预期；端到端草稿管道可运行 |
| **阶段3（第6-9周）** | 实现 Agent 记忆与反思机制；前后端集成；系统端到端测试；人工评估与文档撰写 | 完整原型部署；完成人工评分并分析 |

## 七、评测方法与预期成果

### 7.1 评测指标

- **NLP 模块级**  
  - 分类：Macro-F1、Accuracy；NER：Span-F1；关系抽取：Micro-F1 及各关系 F1  
  - 摘要：ROUGE-1/2/L、BERTScore；检索：Recall@K、MRR、NDCG@10
- **Agent 系统级**  
  - 任务完成率、正确工具选择率、引入反思后报告质量提升幅度（通过人工评分衡量）
- **人工评估**  
  邀请 3-5 名研究生对生成审稿报告按五维 Likert 量表评分：  
  **完整性**（是否覆盖主要审稿要素）、**准确性**（事实/引用是否正确）、**深度**（是否指出方法局限与实验不足）、**建设性**（建议是否具体可操作）、**可读性**（语言是否清晰规范）。

### 7.2 预期成果

1. 可正常运行、交互的原型系统；
2. 四个独立评测达到预设指标的 NLP 模块；
3. 一套面向科研文献审稿的多 Agent 协作工作流；
4. 输出示例：结构化审稿报告、知识图谱、相似论文推荐；
5. 完整的技术文档与答辩演示材料。

## 八、创新点与可行性分析

### 创新点

1. **深度语义与智能编排的交叉**：率先将自研科学文本抽取、生成、检索等 NLP 模块系统集成于 ReAct 驱动的 Agent 协作框架中，形成审稿闭环。
2. **可信生成机制**：约束生成必须以抽取出的实体与关系为锚点，缓解大模型幻觉，增强审稿意见的可追溯性。
3. **反思记忆设计**：通过短期/长期记忆与反思步骤，使 Agent 能在审稿过程中积累“经验”，提升报告完备度。

### 可行性

- **团队能力**：成员已完成 NLP 课程学习，熟悉 transformers 微调和基础 Agent 开发。
- **资源基础**：所需预训练模型与数据集均为开源，GPU 可向学校或云平台申请。
- **时间风险**：四个 NLP 模块可并行推进，Agent 开发与系统集成时间窗口明确，可控性强。

### 风险预案

| 风险 | 应对 |
|------|------|
| 抽取标注数据量小 | 弱监督（远程监督 + 规则标注）为主，少量人工精标 |
| 长论文超 token 限制 | 滑动窗口分块 + 层级编码聚合 |
| Agent 调用链延迟高 | 模块结果缓存 + 独立 Agent 并行调度 |
| 生成内容偏离原文事实 | 生成前注入抽取事实约束，禁止自由编造 |
